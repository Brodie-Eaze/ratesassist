## 1. How to reproduce

```bash
cd apps/web
LOG_LEVEL=fatal RA_TOOL_TRANSPORT=inproc npx tsx scripts/load/load-harness.ts
```

(`RA_TOOL_TRANSPORT=inproc` is also forced inside the harness; `LOG_LEVEL=fatal`
silences the per-call pino lines so the report table is readable — the harness
measures wall-clock latency with `process.hrtime.bigint()` directly, not from
logs. `ANTHROPIC_API_KEY` is unset by the harness to guarantee the MOCK LLM
path.)

Optional knobs (env): `LOAD_PEAK` (default 75), `LOAD_DURATION_MS` (4000),
`LOAD_WARMUP_MS` (750), `LOAD_SWEEP_PARCELS` (5000), `LOAD_COUNCIL` (TPS),
`LOAD_JSON=1` (machine-readable tail).

Determinism: seeded `mulberry32` PRNG for all synthetic data; engine clock
pinned to `2026-05-29`. Re-runs on the same machine are stable to within
event-loop jitter (confirmed across 3 runs; p99s moved < 10%).

## 2. Concurrency model — "projected peak" for a council pilot

A single council rates department running a pilot is **~20–30 officers**. We
size base concurrency at **25 simultaneously-active officers** and apply a **3×
burst** multiplier (Monday-morning arrears run, end-of-quarter reconciliation)
→ **~75 concurrent in-flight operations**. That is the documented peak the
numbers below are measured at.

The harness is **closed-loop**: it holds exactly N operations in flight for a
fixed window; each worker awaits its op, records wall-clock latency, then issues
the next. This measures the latency a real caller sees once the loop is
saturated (queuing + single-core CPU contention included) — a stricter test
than open-loop fire-and-forget.

Three scenarios:

1. **MIXED @ 75** — all paths interleaved in a realistic read-heavy pilot mix
   (pool weights: properties ×3, chat ×2, health/ready/sweep_demo/verify_chain/
   sweep_engine ×1). This is the headline "peak" result.
2. **ISOLATED @ 75** — each path run alone at peak, so a slow path can't hide
   behind fast ones. **This is the distribution the PASS/FAIL verdict is judged
   on** (cleanest per-path p99).
3. **OVERLOAD @ 750** — the cheap read path (`/api/properties`) driven at 10×
   peak to locate the cliff and prove headroom.

### Path → code mapping (what each op actually drives)

| Harness op | SLO path | Real code exercised |
|---|---|---|
| `chat` | `POST /api/chat` | `runChat()` on the **MOCK LLM** path → full intent-router + the multi-tool dispatch loop (2–3 `runTool` calls/turn), with tenant scoping applied. |
| `properties` | `GET /api/properties` | `runTool("list_properties", {limit:50})` — the grounding read. |
| `sweep_demo` | (chat tool-call) | `runTool("find_mining_mismatches", {minSeverity:"low"})` over the demo dataset (~115 parcels) — the sweep as the chat surface calls it. |
| `sweep_engine` | (chat tool-call, stress) | raw `findMismatches(ctx)` at **council scale: 5,000 parcels** — worst-case CPU sweep a large pilot (Kalgoorlie / East Pilbara) triggers. |
| `verify_chain` | `GET /api/audit/verify-chain` | `verifyChain()` from `@ratesassist/audit-core` over a synthetic **1,000-row valid chain** — the route's actual CPU driver (the F-011 note in the route confirms the SHA-256 recompute *is* the latency, not the DB read). |
| `ready` | `GET /api/ready` | `listMcpTools()` — the catalogue probe the readiness handler runs. |
| `health` | `GET /api/health` | the fixed-200 handler body (JSON build, no deps). |

The recovery sweep is not a discrete SLO row; its latency contributes to the
**chat** p99 (it is a tool-call inside a turn). It is judged against the
2,000 ms sweep budget from `scripts/perf-bench.ts` and surfaced separately.

## 5. What this harness does NOT simulate (read before quoting)

These caveats are material. The numbers above measure **RatesAssist's own code
contribution** to latency under concurrency — not end-to-end production latency.

1. **No network. The LLM is the MOCK path; transport is `inproc`.** Real chat
   p99 in production is dominated by the **Anthropic round-trip** (seconds),
   which the harness deliberately does not include. The SLA explicitly
   **excludes** Anthropic latency/outages from the error budget, and the chat
   SLO (p99 < 20 s) is sized to absorb the model RTT. So the 12 ms chat figure
   is "our orchestration + tool dispatch overhead", and the headroom to 20 s is
   the model's budget. **This harness proves our code is not the chat
   bottleneck; it does not validate the model's latency.**
2. **No real DB latency.** `RA_USE_DB` is off. `list_properties` reads the
   in-memory demo store; `verify_chain` hashes an in-memory synthetic chain. The
   route's Postgres read is **not** measured. For `verify-chain` this is
   defensible — the route's F-011 note attributes the latency to the SHA-256
   work (which we *do* measure), and the chain walk uses the indexed
   `(tenant_id, occurred_at, id)` order so the read is ~ms. For
   `/api/properties` a production read adds one PgBouncer-fronted indexed query.
   **A real-DB run requires a provisioned Postgres + `RA_USE_DB=true` and is the
   recommended next step before the SLO is signed off for production tier.**
3. **Single process / single core.** Production runs multiple Fargate/Vercel
   replicas. This is the **per-replica worst case** — if one replica holds
   under SLO at 75 concurrent, the fleet does too. The closed-loop model with
   all CPU work on one event loop is, if anything, pessimistic vs a multi-core
   serverless deployment.
4. **Demo dataset for the read/sweep_demo paths (~115 parcels).** The
   `sweep_engine` path compensates by stressing the real engine at 5,000
   parcels; the demo paths measure dispatch + scoping + serialisation overhead,
   not large-dataset scan cost (which `scripts/perf-bench.ts` already covers).
5. **No middleware / rate-limit / auth-resolver cost.** The harness calls
   `runChat`/`runTool`/`verifyChain` directly, below the Next.js route handler.
   Middleware, the session resolver, and the per-route rate limits add a small
   fixed overhead in production (and the verify-chain rate limit is a *protective*
   cap, not a latency cost).

## 6. Bottlenecks found

**None within the SLO budgets.** No hot-path fix was required; the recovery
engine (`packages/recovery-engine`) and the audit verifier already clear budget
at peak with headroom. The two CPU-bound paths (`sweep_engine` 730 ms,
`verify_chain` 242 ms isolated) are the closest to their budgets but remain
comfortably under, and both are naturally throttled in production (sweeps are
chat-initiated and infrequent; verify-chain is rate-limited 6/min/IP).

Forward-looking note (no change made): if a future pilot council exceeds ~10k
parcels *and* sweeps become more frequent, the synchronous single-core sweep is
the first path to watch — the designed migration is to move `findMismatches`
off the request event loop (worker thread or a BullMQ-style job) so a heavy
sweep cannot stall concurrent reads on the same replica. Not needed at pilot
scale; flagged for the production-tier SLO review.
